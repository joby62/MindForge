技术集成 Owner 职责清单 v0.1

1. 角色定位

技术集成 Owner 不是“最会写代码的人”，也不是“救火程序员”。

这个角色真正负责的是：

把一节一节的 demo，变成同一种系统。

换句话说，他 / 她是把：

  • 世界包装
  • 交互逻辑
  • 规则系统
  • 报错系统
  • 重跑机制
  • 家长证据

从一次性手工活，变成可持续复用骨架的人。

2. 为什么这个角色必须存在

如果没有技术集成 Owner，最常见的问题是：

  • 第一节能跑，但第二节重做一套
  • 每节的命名、状态、报错都不一致
  • Codex 每次都重造轮子
  • 第三节开始维护成本高于开发成本
  • 家长端和孩子端的数据结构对不上

所以这个角色的作用，不是加快编码，而是：

避免 demo 阶段天然走向碎片化。

3. 在最小 demo 阶段，他 / 她到底负责什么

最小 demo 阶段，不要让技术集成 Owner 负责所有技术问题。

只让他 / 她盯住 5 个核心统一层：

  1. 状态系统统一
  2. 错误类型统一
  3. 法则台组件统一
  4. 运行与重跑机制统一
  5. 配置格式统一

这 5 件事是第一版技术骨架。

4. 具体职责清单

A. 规格可实现性审查

在进入 Codex 执行前，审核：

  • 系统结构卡是否真实可实现
  • 错误设计卡是否不是空概念
  • 交互流程是否会造成状态爆炸
  • 成功标准是否可被程序验证

结论必须明确为：

  • 可直接实现
  • 需缩减后实现
  • 当前不建议实现

B. 建立统一的技术骨架

负责定义并维护：

  • 通用模块对象结构
  • 通用错误对象结构
  • 通用运行状态结构
  • 通用回放结构
  • 通用家长证据输出结构

一句话：

不同章节可以换神话皮，但底层骨架不能每次换。

C. 定义复用层，而不是只盯当前节

每做一节，都要判断哪些东西应进入复用层：

  • 通用法则台
  • 通用报错面板
  • 通用模块状态展示
  • 通用二次重跑逻辑
  • 通用证据抽取逻辑

如果没有这个动作，团队就永远只能做 demo，不能做产品。

D. 作为 Codex 的技术闸门

Codex 接收规格后，技术集成 Owner 要判断：

  • 这次是用旧组件拼装，还是要新建组件
  • 哪些需求应该被砍掉
  • 哪些实现路径会破坏长期统一性
  • 哪些地方允许临时 hardcode，哪些绝不能

也就是说：

Codex 负责施工，技术集成 Owner 负责不让施工把地基打坏。

E. 管理技术债

最小 demo 阶段允许少量技术债，但必须被显式记录。

每周至少记录：

  • 哪些地方是临时写死的
  • 哪些状态命名还不统一
  • 哪些配置未来要抽象
  • 哪些逻辑只是为了这节先跑起来

否则 demo 做完，整个系统会变得不可扩展。

5. 不该由技术集成 Owner 承担的职责

为了防止角色膨胀，以下内容不应主要由技术集成 Owner 决定：

  • 世界观是否成立
  • 神话剧情是否吸引孩子
  • 五大主轴教育价值是否成立
  • 家长文案是否有说服力
  • 最终视觉审美是否成立

这些都需要你和元老会拍板。

技术集成 Owner 的边界是：

决定怎么以统一方式实现，而不是决定产品灵魂。

6. 与你的关系

你负责：

  • 愿景
  • 边界
  • 审美
  • 版本优先级
  • 最终拍板

技术集成 Owner 负责：

  • 实现骨架
  • 统一规则
  • 复用结构
  • 技术红线
  • 技术债记录

一句话关系：

你决定做什么，他 / 她决定怎么统一地做。

7. 与 Codex 的关系

技术集成 Owner 不是 Codex 的替代者，而是 Codex 的上游约束层。

技术集成 Owner 给 Codex 的

  • 可执行规格
  • 技术红线
  • 组件复用规则
  • 本轮允许的简化方案

Codex 回给技术集成 Owner 的

  • 页面实现情况
  • 状态流
  • 报错流
  • 分歧点
  • 技术限制

最重要的原则：

Codex 不直接接“世界观愿景”，只接被技术集成 Owner 审过的规格包。

8. 每周工作节奏建议

周一：规格审查

审核：

  • 系统结构卡
  • 错误设计卡
  • 交互流程卡
  • 成功标准卡

输出：

  • 可实现 / 需缩减 / 不建议做

周二：实现骨架拆解

明确：

  • 哪些组件复用
  • 哪些组件新建
  • 哪些逻辑可配置
  • 哪些逻辑先 hardcode

周三 - 周四：盯灰盒落地

重点看：

  • 逻辑是否跑通
  • 状态是否统一
  • 报错是否可观测
  • 重跑是否成立

周五：技术复盘

输出：

  • 本周新增复用层
  • 技术债列表
  • 下周建议
  • 模板化机会

9. 最小 demo 阶段的优先级顺序

技术集成 Owner 在最小 demo 阶段，只按下面顺序排优先级：

第一优先：单节循环跑通

是否能做到：

  • 判断
  • 修改
  • 运行
  • 报错
  • 重跑

第二优先：错误系统统一

第一版至少要统一：

  • 条件错误
  • 顺序错误
  • 权限错误
  • 状态 / 边界错误

第三优先：法则台统一

不要第一节一个法则台，第二节再发明一个。

第四优先：配置格式统一

哪怕先简陋，也要统一。

第五优先：技术债可见

不是立刻消灭，而是必须被记录。

10. 技术集成 Owner 的输出物

这个角色每周至少要产出 4 类文档 / 决策：

  1. 实现方案说明
  2. 组件复用清单
  3. 技术债清单
  4. 结构统一决策记录

如果这些东西没有留下，说明这个角色实际上只是在做普通程序管理。

11. 红线清单

以下情况，技术集成 Owner 应直接叫停或要求返工：

  • 每节独立造一套状态系统
  • 错误类型命名不一致
  • 报错逻辑只能给最终失败，不能给局部状态
  • 法则台交互每节都不一样
  • Codex 在没有成功标准的情况下开始写实现
  • 家长证据结构与孩子端行为结构脱节

这些都是“后期一定炸”的信号。

12. 最小 demo 阶段，谁最适合兼任这个角色

优先顺序建议：

  1. 最懂前端交互，且有系统抽象能力的人
  2. 最能把逻辑讲清楚、并能约束实现的人
  3. 程序里最有“搭骨架”意识的人

如果暂时没有合适人选，短期可以由你指定一位程序 + 你共同承担，但不建议长期空缺。

13. 判断这个角色是否称职的标准

不是看他 / 她写了多少代码，而是看：

  • 第二节是不是比第一节更容易做
  • 错误系统有没有开始统一
  • Codex 接到需求后返工次数是否下降
  • 每节的组件复用率是否上升
  • 技术债是否被记录和控制

如果这些没有改善，说明技术集成 Owner 只是挂名，没有真正发挥作用。

14. 最终原则

技术集成 Owner 的本质工作，不是“把当前这一节做完”，而是：

让下一节不必重新发明。

他 / 她是最小 demo 团队里把“创意”变成“可持续生产”的关键角色。
