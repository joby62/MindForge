单节工业化开发流程 v0.1

—— 神话编程思维产品的最小生产单元

文档定位

这份文档回答的不是“世界观是什么”，而是更落地的问题：

  • 一节内容到底是什么
  • 一节应该如何被稳定地生产出来
  • 团队如何避免每一节都从零开始手工搭建
  • 如何让孩子在一节里真正经历“判断 - 修改 - 重跑 - 调试 - 提炼”
  • 如何让家长看到可信的成长证据

这是一份面向产品、内容、交互、教学、家长端共同对齐的核心项目文档。

1. 为什么先做“单节”而不是先做“一章”

这个项目的真正风险，不是世界观不够大，而是第一步太大。

如果不能把“一节”这个最小单元跑通，那么： - 世界观会变成昂贵包装 - 章节会变成手工定制内容 - 教育目标无法稳定发生 - 家长端价值无法可信沉淀

所以产品开发必须先缩到最小闭环：

先证明一节可以成立，再证明第二节可以复用，再谈一章、一季、一个世界。

2. “一节”的定义

一节不是一段剧情，也不是一个知识点，更不是一题练习。

一节 = 一个孩子可以在 20–30 分钟内完成的最小认知闭环。

这个闭环至少包含： 1. 进入异常 2. 判断真正问题 3. 看清系统结构 4. 修改法则或规则 5. 运行系统 6. 观察可观测错误 7. 局部修复并重跑 8. 带着一条可迁移的方法离开

如果缺少“重跑”和“报错”，它更像剧情解谜； 如果缺少“提炼”，它更像一次性任务； 如果缺少“判断问题”，它更像教程操作。

3. 一节的北极星体验

对孩子

孩子离开一节时，最好留下三种感受： - 我看出来问题真正出在哪 - 我改了规则，世界真的变了 - 我第一次没修好，但我能靠反馈继续修出来

对家长

家长离开一节时，最好能看见三件事： - 孩子不是乱试，而是先判断再动手 - 孩子真的经历了定位问题和调试 - 孩子带走的不是一题，而是一种方法

4. 第一版玩法核心循环

第一版不需要大地图、战斗、开放世界，也不需要复杂数值成长。

第一版必须跑通的只有这一条：

看异常 -> 判问题 -> 拆模块 -> 改法则 -> 跑系统 -> 看报错 -> 再修一次 -> 得到模式句

这是产品本体。

第 1 段：进入异常（2–3 分钟）

目标不是教学，而是让孩子想管这件事。

第 2 段：判断真问题（3–5 分钟）

目标是区分“表面现象”和“真实问题”。

第 3 段：拆解系统（3–5 分钟）

目标是把一团局面变成模块和路径。

第 4 段：修改法则（6–8 分钟）

目标是让孩子真正修改条件、顺序、权限或路径。

第 5 段：运行系统（2–3 分钟）

目标是让孩子看到自己的规则影响了世界。

第 6 段：读取报错并重跑（4–6 分钟）

目标是让孩子形成真正的调试体验。

第 7 段：母纹提炼（2–3 分钟）

目标是带走一条可迁移的模式句，而不是只结束在“成功”。

5. 工业化生产：八个固定工位

以后任何一节，都必须经过下面八个工位。不得跳步。

工位 1：节目标定义

必须写清： - 属于哪一章、哪一季 - 这一节在整章中的位置 - 主打哪 1–2 个主轴 - 核心体验目标是什么 - 系统对象是什么 - 家长端最终要留下什么证据

产物：节目标卡

工位 2：戏剧钩子设计

只回答： - 孩子一进来最先看见什么 - 听见什么 - 哪个角色先说话 - 表面异常是什么 - 为什么孩子会想介入

产物：进入钩子卡

工位 3：系统建模

必须写清： - 系统名称 - 系统模块 - 输入 - 输出 - 条件 - 依赖关系 - 搅局者拨动点

产物：系统结构卡

工位 4：错误反馈设计

必须写清： - 这一节暴露哪几类错误 - 错误如何被孩子观察到 - 错误如何被分类呈现 - 哪些反馈即时出现，哪些延迟出现 - 哪些反馈属于边界 bug

产物：错误设计卡

工位 5：孩子交互路径设计

必须写清： - 孩子先看什么 - 先判断什么 - 可以做哪些操作 - 哪一步必须自己决定 - 哪一步允许给支架 - 第一次运行后如何进入调试

产物：交互流程卡

工位 6：认知校验

必须回答： - 主打主轴是否真的发生了 - 有没有伪主轴 - 孩子是在猜答案还是在定位问题 - 报错是否帮助孩子形成调试思维 - 最后有没有认知抬升

产物：主轴校验卡

工位 7：家长证据提炼

必须沉淀三类证据： - 行为证据：孩子做了什么判断和修改 - 思维证据：体现了哪种能力 - 调试证据：如何面对错误并继续修复

产物：家长证据卡

工位 8：复盘与模板沉淀

必须记录： - 哪一部分开发最费劲 - 哪类错误最好用 - 哪种反馈孩子最容易理解 - 哪一步家长证据最自然 - 这节最终属于哪类模板

产物：节模板归档卡

6. 五类标准节类型

工业化的前提不是复用剧情，而是先复用“节类型”。

A. 看穿问题节

主打：定义问题

孩子主要做： - 区分表象与真因 - 看清输入、输出、限制 - 判断真正问题属于哪一类

适合错误： - 输入错误 - 误判错误 - 表象误导

B. 理顺结构节

主打：拆解任务

孩子主要做： - 分模块 - 排顺序 - 抓关键依赖

适合错误： - 顺序错误 - 依赖错误

C. 立定法则节

主打：建立规则

孩子主要做： - 组织条件 - 设定权限 - 处理例外 - 修正路径

适合错误： - 条件错误 - 权限错误 - 冲突错误

D. 修补裂缝节

主打：调试纠错

孩子主要做： - 看反馈 - 提假设 - 改局部 - 重跑验证

适合错误： - 边界错误 - 状态错误 - 延迟错误

E. 识别母纹节

主打：抽象迁移

孩子主要做： - 对比两个系统 - 看出共性 - 提炼模式句

适合错误： - 不是主打报错，而是主打结构映射

7. 错误系统是产品灵魂

这个项目要想真正成立为“编程思维产品”，就不能只有剧情异常，必须有可观测的错误系统。

核心原则

  1. 任何失败都必须可观测，不能只可感受
  2. 错误必须有类型，不能只是统一失败
  3. 错误必须是局部可见，而不是整章黑箱失败
  4. 系统必须允许“假设 -> 修改 -> 重跑”
  5. 剧情负责让我想修，诊断系统负责让我会修

第一版建议只做四类错误

  • 条件错误
  • 顺序错误
  • 权限错误
  • 状态 / 边界错误

错误的双层语言

底层要有真正的逻辑错误模型，表层要有世界语言。

例如： - 底层：SequenceMismatch - 表层：纹路闭合顺序错误

  • 底层：ConditionMismatch
  • 表层：火脉未满足开启条件

这样既保留世界感，又保留编程思维的可诊断性。

8. 三个核心界面

第一版不需要很多页面，但这三个界面必须成立。

1. 异常现场界面

作用：负责进入感、世界感、情绪钩子。

应呈现： - 场景异常 - 角色反应 - 世界回响 - 初步线索

2. 系统视图界面

作用：把局面变成可分析对象。

应呈现： - 模块 - 状态 - 连接 - 依赖 - 当前异常位置

3. 法则台界面

作用：改规则、运行、看报错、重跑。

应呈现： - 可修改规则 - 运行按钮 - 报错反馈 - 回放 - 二次调整入口

9. 家长端如何看到“实在的成长”

家长端不要只显示： - 完成了哪一节 - 用了多久 - 是否通关

第一版至少要沉淀三种证据：

1. 问题定义证据

例如： 孩子能区分“资源消失”和“资源流动失序”是两种不同问题。

2. 结构拆解证据

例如： 孩子先将问题拆为“火源 - 通路 - 权限 - 落点”四个模块，再开始排查。

3. 调试证据

例如： 第一次运行后，孩子没有停在失败，而是根据模块状态继续调整规则并重跑。

这三类证据，已经足够形成可信的家长感知。

10. 团队会议应先对齐的四件事

在真正开做之前，建议团队先只讨论四件事：

1. 一节的标准定义

  • 时长是否固定在 20–30 分钟
  • 必须包含哪些闭环

2. 八个工位是否认可

  • 是否需要增删
  • 哪些是强制工位

3. 初期五类节类型是否认可

  • 是否足以覆盖第一季
  • 哪类最适合先做

4. 错误类型体系先定到几类

建议第一版只定 4 类，不要无限扩张。

11. 第一版成功标准

第一版不看世界有多大，只看四个问题：

1.

孩子会不会愿意把一节玩完。

2.

孩子会不会在没有直接答案的情况下，真正做出一次“判断 - 修改 - 重跑”。

3.

报错反馈是不是让孩子更接近问题，而不是更困惑。

4.

家长看完后能不能一句话说出： “这节练的是哪种思维能力。”

只要这四条成立，第一版就是成功的。

12. 结论

这套产品要想成立，首先要证明的不是世界观，而是生产线。

先证明一节可以被稳定生产，且不是靠运气。

只要“看异常 -> 判问题 -> 改法则 -> 跑系统 -> 看报错 -> 再修一次”这条最小循环成立， 后面的世界、角色、章节、希腊主线、家长增长、工业化模板，才有真正往上搭的基础。

从这个角度看，项目当前最重要的工作，不是继续扩概念，而是把一节做硬、做清楚、做成可复制的方法。
