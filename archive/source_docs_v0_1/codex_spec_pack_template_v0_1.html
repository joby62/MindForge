Codex 需求规格包模板 v0.1

1. 文档定位

这份模板用于把一个“单节开发需求”压缩成 Codex 可以稳定执行的规格包（Spec Pack）。

目标不是把愿景讲得更完整，而是把需求讲得更精确，让 Codex 不再接收模糊需求，不再在实现中替团队做关键产品判断。

一句话原则：

Codex 不接“我们想做什么”，只接“这一节到底怎么跑”。

2. 适用范围

适用于以下类型的交付：

  • 单节灰盒 demo
  • 单节正式实现
  • 单节改版迭代
  • 报错系统补完
  • 交互流程重构

不适用于以下类型的输入：

  • 纯世界观讨论
  • 纯剧情方向发散
  • 没有系统对象的概念需求
  • 没有成功标准的探索性任务

3. 规格包的最小组成

每次交给 Codex 的规格包，至少必须包含以下 4 份核心材料：

  1. 系统结构卡
  2. 错误设计卡
  3. 交互流程卡
  4. 成功标准卡

建议补充以下 3 份辅助材料：

  1. 节目标卡
  2. 世界语言与文案约束卡
  3. 明确不做项清单

如果核心 4 卡不齐，原则上不进入 Codex 执行阶段。

4.3 错误设计卡

这是规格包的灵魂层。

  • 本节会暴露哪几类错误
  • 每类错误的底层类型名
  • 每类错误的世界化表述
  • 错误如何被孩子观察到
  • 错误会在哪个节点暴露
  • 哪些错误是第一次运行就出现，哪些是边界 bug

  • 错误必须可分类
  • 错误必须能帮助孩子缩小范围
  • 错误不能只表现为“最终失败”

  • 不能把所有关键判断都变成自动提示
  • 不能让孩子只靠点下一步推进

4.5 成功标准卡

这一张决定 Codex 何时可以认为“做完了”。

必须写成可验收句子，不能写抽象目标。

模板：

成功标准：
1. 孩子必须能完成至少 1 次问题归类判断。
2. 孩子必须能完成至少 2 次规则修改。
3. 系统必须能暴露至少 2 类不同错误。
4. 第二次运行后必须看到局部修复结果。
5. 家长端必须自动生成 3 条成长证据。

禁止写法：

  • 更顺畅
  • 更有 Disney 感
  • 更像冒险
  • 更适合孩子

这些都不是验收标准。

4.7 明确不做项清单（建议）

这张卡非常重要，用来控制实现膨胀。

模板：

本节明确不做：
- 不做大地图探索
- 不做额外支线
- 不做第二套法则台
- 不做动态数值成长
- 不做语音演出

Codex 一旦没有“不做项”，就容易过度发挥或误判版本边界。

5. Codex 执行原则

原则 1：先灰盒，后包装

第一轮只做：

  • 状态是否能跑
  • 规则是否能改
  • 错误是否能出
  • 重跑是否成立

第二轮再补：

  • 世界文案
  • 视觉包装
  • 动效表现
  • 神话风格化

原则 2：发现多条合理实现路径时，必须停下来回报

如果 Codex 发现 2 条以上都讲得通的实现方案，不能自行拍板，必须回报给人类决策层。

建议固定提示语：

如果实现中出现两条以上合理路径，请停止扩写，并回报：
1. 路径 A 的优点 / 风险
2. 路径 B 的优点 / 风险
3. 你建议的默认路径

原则 3：先满足成功标准，再谈扩展

只要成功标准没全部满足，任何额外优化都不进入本轮。

6. 交付给 Codex 前的人工检查清单

在规格包交给 Codex 之前，人工必须确认：

  • 这一节的系统对象是否清楚
  • 这一节的错误类型是否清楚
  • 孩子到底要自己判断什么是否清楚
  • 是否存在至少一次二次修复
  • 成功标准是否都是可验收句子
  • 是否已经写明不做项

只要有一项答不上来，就不要进入执行。

7. Codex 回传物的标准格式

Codex 不应只回传“代码完成”。

建议回传以下 5 项：

  1. 页面清单
  2. 状态流说明
  3. 错误流说明
  4. 需要人工拍板的分歧点
  5. 当前版本与成功标准的对照表

8. 最小示例（简版）

节名称

《雷霆工坊的第三道火》单节版

系统对象

雷霆锻造系统

模块

  • 引火
  • 聚纹
  • 锻打
  • 冷却
  • 绑定

错误类型

  • 顺序错误：冷却过早
  • 状态错误：第三道火未稳定
  • 前置条件错误：绑定未满足

孩子操作

  • 判断问题不在前半段
  • 调整第三道火触发条件
  • 修改冷却顺序
  • 重跑一次

成功标准

  • 孩子完成至少一次问题归类判断
  • 暴露至少两类错误
  • 孩子完成至少两次修改与一次重跑
  • 家长端生成三条成长证据

9. 最终原则

这份模板的底层目标只有一个：

把“模糊愿景”压缩成“可执行规格”。

当 Codex 接到的不是“做一个好玩的神话编程体验”，而是一个结构清楚、错误清楚、成功标准清楚的 Spec Pack 时，团队才真正开始拥有工业化生产能力。
