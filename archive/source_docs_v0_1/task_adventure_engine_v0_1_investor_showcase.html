01 / Definition

任务冒险引擎定义稿 v0.1

1.1 项目的一句话定义

任务冒险引擎是一种以任务冒险为驱动、以角色身份为入口、以系统反馈为支架、以编程思维五大主轴为骨架的儿童互动成长引擎。

对孩子而言，它表现为一个可持续进入的世界：孩子接手局面、看清问题、设计规则、修复异常、提炼规律，并在连续反馈中感知自己正在变强。对家长而言，它表现为一种可解释、可追踪、可累积的能力成长系统。

它不是什么

不是“更花哨的在线编程课”，也不是“学习标签下的小游戏”。如果一端过分偏教育，孩子不会留下；如果一端过分偏娱乐，家长不会续费。

它真正是什么

一种新的成长媒介：让孩子在世界化、身份化、反馈化的冒险系统中，自然习得定义问题、拆解任务、建立规则、调试纠错、抽象迁移的能力。

1.2 为什么“Disney 式体验”适合作为承载方式

所谓“Disney 式”，并不等于表层视觉卡通化，而是指一种高度完整的吸收机制：孩子先被世界吸引，再在世界中认领位置，然后在真实行动与连续反馈中获得成长感，最终形成对这个宇宙的情感归属。

而编程思维本身，恰恰就是“在规则世界里行动，并不断根据反馈修正策略”的能力。因此，冒险体验不是编程思维的装饰，而是它最自然的承载方式之一。

1.3 我们真正训练的能力是什么

表面上叫“编程思维”，但本质上更接近一种系统化理解世界的能力：把混乱问题转化为清晰结构，把模糊世界转化为可理解规则，把错误结果转化为可修正路径。

系统思维 结构化表达 规则建模 问题求解 反馈修正 模式迁移

1.4 五大主轴不是“内容模块”，而是世界中的五种核心行动

主轴在业务中的本质含义对孩子的真实体验
定义问题在信息不完整、目标不完全明确的局面中识别真正的问题与约束。我先看清局面真正卡在哪里。
拆解任务把复杂问题拆成可执行、可排序、可组织的局部问题。我能把一团乱整理成行动路径。
建立规则把主观意图转化成可以被世界执行的秩序与系统。我设计的规则真的让世界运转起来。
调试纠错在结果与预期不一致时，通过观察、猜测、验证与修正推动模型升级。系统出问题了，但我能找出来并修好。
抽象迁移从单次任务中提炼出一类问题的共通结构与可复用方法。我发现了隐藏在不同任务背后的同一种规律。

1.5 业务核心的最终表达

把思维训练，转化为孩子愿意主动反复进入的冒险体验；再把这种体验中的能力增长，转化为家长能够稳定感知的成长证据。

这一定义意味着：具体产品形态、交互方式、任务包装、AI 角色，都是愿景的延伸；但业务本体始终是“投入机制 + 成长确定性”。
02 / Design Principles

任务冒险引擎设计原则 v0.1

设计原则的作用，是在后续所有产品、内容、交互、AI 与家长端决策中，持续防止项目回滑为课程包装、题库换皮或空心娱乐。

2.1 设计总纲

我们的产品不是“把学习做得更有趣”，而是“把成长做成可投入的世界”。这意味着：任务首先必须作为值得参与的局面成立，然后能力训练才在其中发生。

世界成立孩子感到这里是一个持续存在、值得进入的地方。
→
任务成立局面真实、必要、可介入，且必须通过思考推进。
→
能力发生五大主轴被真实调用，而不是被讲解灌输。
→
成长可见反馈与过程痕迹被沉淀为能力证据。

2.2 八条硬约束

原则约束内容业务意义
任务优先任务必须先是值得参与的局面，再是能力训练的载体。防止产品退化为练习题换皮。
主轴隐入行动五大主轴必须藏在行动中，不能主要靠讲解出现。保证孩子是在“被调用能力”，不是“被灌输概念”。
身份优先于学生感孩子进入世界时首先获得身份感，而不是被动学习者定位。提升主动性、尊严感与能力欲。
挑战真实，失败不羞辱问题必须是真问题，但失败必须被解释为“系统还没调好”。让调试成为成长入口，而不是挫败来源。
反馈面向能力增长反馈不只告诉孩子对错，而要指出他在哪个思维动作上变强。建立长期留存所需的真实能力感。
保留过程痕迹每个任务都必须沉淀理解、拆解、版本与修正痕迹。为复盘、家长感知和个性化成长提供基础。
AI 只增强思考AI 可以提问、支架、陪调试，但不能跳过核心思维动作。避免系统被“代做式 AI”掏空。
双边同时成立所有设计都必须同时服务于孩子的投入感与家长的成长感。确保长期留存与续费逻辑闭环。

2.3 三条高风险误区

误区一：更花哨的在线课

如果“老师讲、孩子听”仍是主形态，即便加上动画、闯关、徽章，也只是旧逻辑的新皮肤，难以形成真正的自驱式参与。

误区二：学习版小游戏

如果孩子留下的只有开心，没有思维成长的稳定沉淀，家长最终会认定它只是娱乐，而非高价值教育产品。

误区三：低幼化陪伴

7–13 岁用户需要的是挑战、尊重、身份与能力感，不是被哄。语气、视觉与奖励机制一旦过度低幼化，产品会迅速失去吸引力。

设计原则不是文案口号，而应成为评审机制：任何新功能、新任务、新交互都必须接受“是否增强世界成立、任务成立、能力发生、成长可见”的检验。
03 / Structural Model

任务冒险引擎结构模型 v0.1

结构模型的目标，是把世界、身份、任务、思维成长与家长感知组织成同一个系统，而不是互相断裂的几块内容。

3.1 五层嵌套系统

世界层提供长期投入的意义容器与宇宙连续性。
身份层给孩子一个非学生化的位置与行动授权。
任务层以事件单位触发真实思维动作。
思维层五大主轴在行动中被调用并循环升级。
证据层把成长沉淀为孩子、家长与系统都可读取的痕迹。

3.2 五层的业务职责

层级核心职责如果缺失会发生什么
世界层让任务具有连续性、必要性与归属感。任务会沦为孤立练习，长期留存变弱。
身份层让孩子以参与者而非被训练者的姿态进入。主动性下降，学生感增强。
任务层以局面而非题目驱动孩子介入与选择。能力动作无法真实发生。
思维层触发定义、拆解、规则、调试、迁移的动态循环。产品会只剩剧情或操作，而失去认知价值。
证据层把过程转化为可复盘、可解释、可持续的成长轨迹。家长看不清价值，系统也难做个性化升级。

3.3 一次完整任务的标准运行流程

任务进入孩子接收到一个局面，而不是一条命令。
情境理解识别目标、条件、约束与缺失信息。
方案拆解组织步骤、模块、顺序与关键路径。
规则表达把方案转成可执行结构。
系统运行世界对孩子的设计做出真实反馈。
异常反馈呈现成功点、失败点与未覆盖情形。
调试修正观察、猜测、修改、再验证。
模式提炼把本次任务提升为可迁移的方法。
成长沉淀自动形成版本、痕迹与能力记录。

3.4 三种界面，本质是一件事的三种投影

孩子界面

孩子看到的是世界、身份、任务、反馈与升级；他感受到的是“我在这里越来越会解决复杂问题”。

家长界面

家长看到的是作品、过程痕迹、能力变化与下一步方向；他感受到的是“孩子的思维正在被稳定塑造”。

系统界面

系统内部处理的是五大主轴调用情况、错误模式、支架介入、成长画像与个性化推荐。也就是说，孩子端与家长端并不是两套逻辑，而是同一条任务链的不同视角。

3.5 AI 在结构中的正确位置

AI 不在中心，但在关键节点提供支架。它最适合出现在理解节点、拆解节点、调试节点与迁移节点——也就是“思维即将发生，但还不稳”的位置。AI 的职责是提问、镜像、提示与陪同，而不是代替孩子完成核心任务。

结构模型最终指向一个闭环：投入 → 行动 → 反馈 → 修正 → 提炼 → 证据 → 再投入。这是项目真正可积累的循环资产。
04 / Experience Goals

任务冒险引擎体验目标 v0.1

一个产品真正成立，不是因为逻辑完整，而是因为它能稳定地产生某种体验。对 7–13 岁而言，体验不是装饰，而是驱动一切发生的第一引擎。

4.1 体验总目标

我们不是要让孩子“上课很开心”，而是要让孩子“在变强的感觉里上瘾”。短暂的热闹不能支撑长期投入；能支撑长期投入的，是能力成长带来的真实爽感。

真正要制造的，不是浅层愉悦，而是“我原来看不懂、做不好、修不动的事情，现在我能看清、组织、设计并修好了”的认知成就体验。

4.2 7–13 岁用户真正会被什么吸引

A
被当成有能力的人
他们会敏锐感知自己是否被尊重、是否被当成真的能解决复杂事情的人。
B
真实挑战而非幼稚难度
他们不怕难，只怕低水平重复与假挑战。
C
选择权与判断空间
他们需要在任务中做出自己的判断，而不是一路被系统牵着走。

4.3 我们必须稳定地产生的八种核心体验

体验名称孩子的心理表述与五大主轴的关系
被召唤感这件事需要我来处理。提升任务必要性与主动介入。
身份感我不是学生，我是某种能解决问题的人。强化非学生化投入姿态。
看清感我把原本混乱的局面看懂了。直接驱动定义问题。
组织感我把一团乱整理成了有路径的结构。直接驱动拆解任务。
掌控感我设计的规则真的让世界运行起来了。直接驱动建立规则。
修复感出问题了，但我能找出来并修好它。直接驱动调试纠错。
发现规律感我看出来这和上次那个其实是同一类问题。直接驱动抽象迁移。
成长可见感我和以前真的不一样了。把单次任务提升为长期留存动力。

4.4 什么样的爽感是对的

正确的爽感：认知型爽感

看懂复杂局面的爽感、把混乱整理成结构的爽感、让规则跑起来的爽感、修掉 bug 的爽感、发现共性模式的爽感，以及“我真的在变强”的爽感。

错误的爽感：替代型爽感

纯收集、纯装饰、纯动画奖励、纯外部排名、纯剧情消费。如果这些成为主要驱动力，系统就会逐步背离自己的教育本体。

4.5 一次高质量任务应有的体验曲线

高质量任务不应一路平铺，而应形成明确的情绪与认知起伏：被卷入 → 产生不确定 → 逐步看清 → 开始建构 → 遇到阻碍 → 调试回升 → 结果兑现 → 认知抬升。

尤其需要强调：高潮不只在“成功通关”，也必须在“看懂”和“修好”这两个点上。因为那才是编程思维真正让孩子着迷的瞬间。

4.6 最终体验表达

孩子在这里感受到的，不是“我正在学习”，而是“我正在作为一个越来越厉害的问题解决者，进入世界、看清局面、建立规则、修好系统，并一次次确认自己真的变强了”。

当这种体验稳定成立时，孩子获得的是持续回来的理由，家长获得的是稳定续费的理由，项目获得的则是真正可复利的增长引擎。
Closing Note

结语：资金视角下的价值判断

从资本视角看，这个项目的潜在价值不在于“又一套儿童编程课”，而在于它试图建立一种更高层的业务形态：一种兼具儿童内容吸引力、认知成长可验证性、AI 支架能力和长期 IP 扩展潜力的互动成长系统。

如果上述四份 v0.1 文件成立，那么后续具体产品形态——任务原型、交互系统、角色体系、家长端、AI 引擎、商业模式——都将围绕一个统一内核展开，而不是拼凑式演化。

本轮展示的核心结论
先定义业务本体，再生长产品形态。只有先把“为什么孩子会沉浸、为什么家长会续费、为什么这不是旧教育逻辑的改装版”讲清楚，后面的产品设计与资本叙事才会真正站得住。
