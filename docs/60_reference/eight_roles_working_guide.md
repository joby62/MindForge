# 八角色分工说明

版本：v0.1 整理版  
用途：作为“任务冒险引擎”单节工业化开发时的角色分工说明

## 1. 先讲清楚一件事

这里的“八角色”不是八个编制岗位，而是八个不能缺的功能位。

这意味着：

- 一个人可以兼任多个角色
- 一个小团队也可以把八个功能都覆盖
- 重点不是人数够不够，而是这些关键判断有没有被做过

如果八个功能位缺失，最常见的后果是：

- 只有创意，没有可执行系统
- 只有故事，没有错误设计
- 只有页面，没有思维动作
- 只有孩子体验，没有家长证据
- 只有本节能跑，没有复用资产

所以八角色的本质，不是为了“分工好看”，而是为了让单节开发变成一条稳定生产线。

## 2. 八角色总览

1. 节目标官：定义这节为什么存在
2. 戏剧钩子官：定义孩子为什么想进去
3. 系统建模官：定义背后到底是什么系统
4. 错误反馈官：定义系统错时孩子到底看见什么
5. 交互路径官：定义孩子怎么走完这一节
6. 认知校验官：定义这节是不是真的在练编程思维
7. 家长证据官：定义家长到底能看到什么成长
8. 模板沉淀官：定义这一节做完后留下什么可复用资产

## 3. 每个角色到底负责什么

### 3.1 节目标官

这个角色负责把“一节”从一个模糊想法压成一个明确目标单元。

要回答的问题：

- 这节在整章里处于什么位置
- 这节主打哪 1-2 个主轴
- 孩子离开时应该带走什么方法
- 这节对世界推进有什么作用
- 这节对家长端要留下什么证据
- 这节明确不做什么

产物：

- 节目标卡

最重要的边界：

- 可以拍板这节值不值得做、主打什么、不做什么
- 不能拍板错误怎么呈现、交互怎么走、系统怎么拆

如果这个角色没做好，后面所有工位都会在错误的目标上用力。

### 3.2 戏剧钩子官

这个角色负责“第一分钟的进入感”。

要回答的问题：

- 孩子第一眼看到什么异常
- 第一耳听到什么
- 哪个角色先说话
- 第一波误导信息是什么
- 为什么孩子会想继续往下看
- 这节整体情绪是什么色调

产物：

- 进入钩子卡

最重要的边界：

- 可以拍板开场异常、开场角色反应、第一印象
- 不能拍板系统根因、错误结构、底层规则逻辑

它的标准不是“剧情写得美”，而是“孩子 2 分钟内会不会想继续管这件事”。

### 3.3 系统建模官

这个角色负责把故事压成一个可执行系统。

要回答的问题：

- 这节的系统对象是什么
- 系统有哪些模块
- 输入输出是什么
- 条件是什么
- 依赖关系是什么
- 状态层怎么定义
- 搅局点到底改了什么

产物：

- 系统结构卡

最重要的边界：

- 可以拍板模块划分、输入输出、依赖和根因层
- 不能拍板角色台词、UI 视觉、家长报告话术

它的完成标准很朴素：程序和交互人员能不能看懂这个系统到底怎么跑、哪里会错、哪里能改。

### 3.4 错误反馈官

这个角色负责把“出错”变成真正的学习入口，而不是一次失败结算。

要回答的问题：

- 这节暴露哪些错误
- 这些错误怎样被孩子看到
- 第一次运行后哪里应该故意不完美
- 反馈如何帮助孩子缩小问题范围
- 哪些错误是首次必现，哪些是边界 bug
- 世界语言怎么说才既有世界感又可诊断

产物：

- 错误设计卡

最重要的边界：

- 可以拍板错误类型、暴露方式、第一次运行后的残留问题
- 不能拍板主打主轴、是否立项、家长怎么解读

如果这个角色做坏了，孩子就只会“失败”，不会“调试”。

### 3.5 交互路径官

这个角色负责把整节体验设计成一条真正会发生判断和修改的路径。

要回答的问题：

- 孩子先看什么
- 先判断什么
- 每一步能做什么
- 支架在哪里给
- 第一次运行在哪里发生
- 重跑从哪里进入
- 结束条件是什么

产物：

- 交互流程卡

最重要的边界：

- 可以拍板操作顺序、入口层级、重跑触发方式、哪些判断必须由孩子亲自做
- 不能拍板系统逻辑是否合理、错误类型是否成立、叙事目标是什么

它要防止的是：页面虽然很多，但孩子其实只是按流程点完。

### 3.6 认知校验官

这个角色专门负责识别“伪编程思维”。

要回答的问题：

- 这节主打主轴是否真的发生
- 哪一步真的有判断
- 错误反馈是否真的促进了思维
- 认知负担是否匹配年龄带
- 这一节结束时有没有认知抬升

产物：

- 主轴校验卡

最重要的边界：

- 可以一票否决伪主轴、伪调试、伪判断
- 不能拍板世界观风格、UI 审美、角色是否讨喜

这个角色最关键的一句判断是：

如果把剧情皮拿掉，这一节还剩不剩真实的思维动作？

### 3.7 家长证据官

这个角色负责把孩子端真实发生的行为翻译成家长端价值。

要回答的问题：

- 哪些行为可以作为证据
- 哪些思维变化可以被短句准确表达
- 哪些调试过程值得被家长看到
- 怎么表达才可信，而不是鸡汤

产物：

- 家长证据卡

最重要的边界：

- 可以拍板呈现哪三条证据、如何表达更可信
- 不能拍板节内机制、系统结构、错误模型

它的标准不是“写得感人”，而是家长看完能不能复述：这节到底练了什么，而且孩子是怎么练到的。

### 3.8 模板沉淀官

这个角色负责把“一次做成”变成“以后更容易做成”。

要回答的问题：

- 这节属于哪种节类型
- 哪些模块值得复用
- 哪些错误模板值得复用
- 这次踩了什么坑
- 哪些做法以后必须沿用
- 哪些地方以后禁止再犯

产物：

- 模板归档卡

最重要的边界：

- 可以拍板哪些资产进入模板库、哪些设计以后必须沿用
- 不能拍板单节是否立项、叙事方向、审美风格

如果这个角色缺席，团队就会一直做 demo，而不是做产品。

## 4. 八角色如何接力

更准确地说，这不是八个孤立岗位，而是一条接力链：

1. 节目标官定目标边界
2. 戏剧钩子官定进入张力
3. 系统建模官定系统骨架
4. 错误反馈官定调试入口
5. 交互路径官定体验路径
6. 认知校验官做真实性检查
7. 家长证据官做价值翻译
8. 模板沉淀官把结果沉淀成资产

如果顺序乱了，通常会出现两种问题：

- 先做页面，后补目标，导致返工
- 先写剧情，后补系统，导致结构发虚

## 5. 八角色对应的八张卡

为了让分工真正落地，建议每个角色至少留下一个明确产物：

1. 节目标官 -> 节目标卡
2. 戏剧钩子官 -> 进入钩子卡
3. 系统建模官 -> 系统结构卡
4. 错误反馈官 -> 错误设计卡
5. 交互路径官 -> 交互流程卡
6. 认知校验官 -> 主轴校验卡
7. 家长证据官 -> 家长证据卡
8. 模板沉淀官 -> 模板归档卡

这些卡的价值在于：

- 让多人协作时边界清楚
- 让 Codex 接收的是可执行对象，而不是空泛愿景
- 让试玩和复盘时有可检查的中间层
- 让模板库真正积累下来

## 6. 管理上怎么用这八角色

小团队里，不需要为了“角色完整”强行扩编，而应该这样用：

- 先确认八个功能位有没有被覆盖
- 再确认每个功能位的产物是否留下
- 最后确认谁拥有哪类拍板权

一个常见的兼任方式是：

- 产品负责人兼任节目标官 + 认知校验官
- 内容负责人兼任戏剧钩子官 + 家长证据官
- 系统策划兼任系统建模官 + 错误反馈官
- 交互负责人兼任交互路径官
- 技术 / 产品复盘负责人兼任模板沉淀官

这只是示例，关键不在岗位名称，而在功能不要丢。

## 7. 三条红线

如果团队在执行中出现以下情况，说明八角色机制实际上失效了：

1. 一节没有明确主打主轴，却已经开始做页面
2. 报错只是最终失败提示，没有可观察、可定位、可缩小范围的信息
3. 一节做完后没有留下模板、错误复用项和踩坑记录

## 8. 最终理解

八角色分工的真正意义，不是把流程变得更官僚，而是把“创意驱动的内容生产”变成“可复盘、可复用、可交给 Codex 执行”的产品生产方法。

一句话总结：

八角色不是八个人，而是八个不可缺的判断环节；少一个，单节就容易重新滑回灵感式开发。
