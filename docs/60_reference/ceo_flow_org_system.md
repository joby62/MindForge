# CEO 视角流程分工设计

版本：v0.1  
定位：从公司级经营和产品工业化视角，为“任务冒险引擎”设计一套可落地的流程分工系统

## 1. 从哪里开始

从业务闭环开始，不从岗位名称开始。

先回答四个问题：

1. 我们到底要稳定产出什么
2. 这个结果的最小生产单元是什么
3. 这个单元从想法到交付要经过哪些不可跳过的步骤
4. 每一步谁提案、谁拍板、谁执行、谁验收

如果这四个问题没回答清楚，任何组织图都只是装饰。

对当前项目而言，答案应当是：

- 稳定产出物：可复用、可验证、可持续扩张的单节任务冒险体验
- 最小生产单元：一节
- 核心业务闭环：孩子愿意进入 -> 真实发生思维动作 -> 家长看见成长证据 -> 团队沉淀模板资产
- 管理目标：把单节从灵感式开发变成工业化生产

## 1.1 当前已拍板经营判断

以下内容已明确，不再作为开放讨论项：

- 当前阶段主战场：单节优先，不并行扩张世界观、增长或商业化主线
- 当前最重要的产品事实：单节闭环必须先跑通，这是孩子兴趣和留存的基石
- 当前最重要的商业事实：家长证据必须清楚可解释，这是家长端价值和付费正当性的基础
- 当前一级战略升级：`Vibe Coding for Kids` 已正式成立，单节闭环将以“对话驱动”为主要升级方向
- 当前阶段的第一梯队指标：单节闭环跑通率、家长证据可解释率
- 当前阶段的第二梯队指标：返工率，后续再优化
- 组织结构：8 个功能位全部保留，不压缩
- 关键岗位状态：世界观总负责人已有人，技术集成 Owner 已有人
- 分歧处理规则：只要出现两条以上都合理、且会影响产品方向的路径，默认停下来回报，不自行替上层拍板

这一组判断意味着：

- 先证明一节能让孩子走完
- 再证明一节能让家长看懂价值
- 再证明这件事不是题目驱动，而是对话驱动地发生
- 其他优化都不能抢在这两件事前面

## 2. 公司级组织设计的总原则

### 原则 1：先定结果，再定流程

组织不是为了公平分工，而是为了稳定产出结果。

### 原则 2：先定流程，再定角色

先把“结果如何发生”画出来，再决定哪些功能位必须存在。

### 原则 3：先定角色，再定人

一个人可以兼任多个功能位，但功能位不能缺。

### 原则 4：每个结果只能有一个第一责任人

可以多人参与，但不能多人共同主责。

### 原则 5：每个环节必须留下明确产物

没有产物就没有交接，没有交接就没有规模化。

## 3. 这家公司应该先搭哪条主链

我会先搭一条“单节生产主链”，因为这是当前业务最小闭环，也是未来所有规模化的基础。

主链如下：

1. 业务目标定义
2. 单节立项
3. 进入感设计
4. 系统建模
5. 错误系统设计
6. 交互路径设计
7. 认知真实性校验
8. 家长证据翻译
9. 技术集成与 Codex 执行
10. 灰盒试玩
11. 正式验收
12. 模板沉淀与复盘

这条链是公司当前最重要的生产线。  
如果这条线不稳，世界观、角色、品牌、增长、商业化都会建立在松土上。

现阶段不并行建设第二主链。  
单节生产线就是当前公司的主线工程。

## 4. 先搭三层，不先搭大组织

在当前阶段，我不会先铺开大公司架构，而会先搭三层。

### 4.1 第一层：方向与拍板层

负责三件事：

- 定义业务北极星
- 决定版本优先级
- 处理关键分歧拍板

建议角色：

- CEO / 产品总负责人
- 产品灵魂位 / 世界观总负责人
- 技术集成 Owner

这一层不负责日常细节执行，但要对“做什么”和“什么先做”负最终责任。

当前状态：

- 世界观总负责人已明确
- 技术集成 Owner 已明确
- 因此方向层现在可以进入真正拍板，而不是继续讨论岗位设置

### 4.2 第二层：单节生产层

这是最核心的一层，直接负责把一节做出来。

建议按八个功能位运转：

1. 节目标官
2. 戏剧钩子官
3. 系统建模官
4. 错误反馈官
5. 交互路径官
6. 认知校验官
7. 家长证据官
8. 模板沉淀官

这一层负责的不是空谈概念，而是把一节变成可执行规格。

### 4.3 第三层：实现与验证层

负责把规格变成可运行原型，并做试玩验证。

建议角色：

- 技术集成 Owner
- Codex / Agent
- 前端 / 交互实现
- 后端 / 状态与数据结构实现
- 试玩观察与反馈记录人员

这一层的职责不是决定产品灵魂，而是把灵魂落成系统，并把问题带回来。

## 5. 先定谁能拍板什么

组织失控，通常不是因为人不努力，而是因为拍板权不清楚。

我会先把拍板权切清楚。

### CEO / 产品总负责人拍板

- 北极星目标
- 阶段优先级
- 单节是否立项
- 哪条路径作为默认产品路径
- 哪些需求本轮直接砍掉

### 世界观 / 内容总负责人拍板

- 世界风格
- 角色体系
- 情绪方向
- 长线章节推进关系

### 节目标官拍板

- 这节主打什么
- 这节不做什么
- 这节对孩子和家长的核心价值

### 系统建模官拍板

- 系统模块如何拆
- 输入输出和状态如何定义
- 搅局点在哪里

### 错误反馈官拍板

- 错误类型
- 错误暴露方式
- 首次运行的不完美点

### 交互路径官拍板

- 操作顺序
- 判断点位置
- 运行点和重跑点

### 认知校验官一票否决

- 伪主轴
- 伪调试
- 伪判断

### 技术集成 Owner拍板

- 哪些逻辑进入统一骨架
- 哪些组件复用
- 哪些地方可以临时 hardcode
- 哪些实现会破坏长期统一性

### CEO 只在两种情况下介入细节

- 路径出现实质性分歧
- 某一环节开始破坏北极星

### 关于多路径分歧的硬规则

只要出现两条以上都讲得通的路径，且差异会影响以下任一项，就必须停下来回报：

- 主打主轴
- 系统结构
- 错误模型
- 家长证据
- 长期复用方式

只有纯实现细节、不影响上述五项时，技术集成 Owner 才可以自行拍板。

## 6. 先定义标准输入输出，而不是先聊创意

每一节都应该按固定接口进入系统。

### 输入物

1. 节目标卡
2. 进入钩子卡
3. 系统结构卡
4. 错误设计卡
5. 交互流程卡
6. 主轴校验卡
7. 家长证据卡
8. 模板归档卡

### 交给 Codex 前的最小执行包

至少四张卡必须齐：

1. 系统结构卡
2. 错误设计卡
3. 交互流程卡
4. 成功标准卡

如果连这四张卡都没有，不允许进入实现。

补充要求：

- 家长证据不能只在最后包装阶段补写
- 从单节立项开始，就必须回答“这节最终要让家长看见哪三条证据”
- 如果家长证据始终说不清，这节即使孩子端体验不错，也不能视为完全成立

## 7. 公司该怎么开会

我不会让团队一直开“泛讨论会”，而是把会议拆成固定节奏。

### 周一：立项与规格审查会

只审五件事：

- 这节值不值得做
- 主打主轴是否清楚
- 系统对象是否成立
- 成功标准是否可验收
- 家长证据是否能被提前说清

输出：

- 通过
- 缩减后通过
- 打回重写

### 周二：实现拆解会

只拆三件事：

- 哪些能力复用旧骨架
- 哪些模块需要新做
- 本轮哪些东西明确不做

输出：

- 可执行任务包
- 技术红线
- Codex 输入包

### 周三到周四：灰盒落地

重点只看：

- 状态是否跑通
- 规则是否可改
- 报错是否可见
- 重跑是否成立

### 周五：试玩与复盘

重点只问：

- 孩子有没有真实判断
- 报错有没有帮助孩子缩小范围
- 家长证据能不能一句话讲明白
- 哪些结构进入模板库

## 8. 这家公司当前最重要的 KPI 不是什么

现在阶段，我不会先盯：

- 页面数
- 内容总量
- 地图规模
- 角色数量
- 世界观展开速度

这些都不是当前阶段最关键的经营指标。

## 9. 当前阶段真正该盯的 KPI

当前阶段先盯以下四个指标，其中前两个属于第一梯队：

1. 单节闭环跑通率
2. 家长证据可解释率
3. 错误系统统一率
4. 第二节相对第一节的生产提速幅度

返工率保留为后续优化指标，当前阶段不放入第一梯队。

如果单节闭环跑不通，说明孩子端基础没成立。  
如果家长证据讲不清，说明商业价值表达没成立。  
如果第二节没有比第一节更容易做，说明公司还没有开始形成真正的生产能力。

## 10. Codex 应该放在什么位置

Codex 不是项目 owner，也不是业务拍板人。

Codex 最合适的位置是：

- 规格执行层
- 灰盒快速搭建层
- 统一结构落地层
- 分歧显性回报层

Codex 不应该直接接：

- 模糊愿景
- 纯世界观口号
- 没有错误模型的需求
- 没有成功标准的任务

所以 CEO 要做的一件关键事，不是“让 Codex 更聪明”，而是“让输入给 Codex 的规格更干净”。

## 11. 组织设计里最容易犯的五个错误

### 错误 1：先按岗位招人，再想流程

结果是每个人都很忙，但系统没有主线。

### 错误 2：让所有人都参与所有决策

结果是没有真正负责人，只有无穷争论。

### 错误 3：把内容生产和系统生产混在一起

结果是每做一节都像重新创业。

### 错误 4：太早追求大而全

结果是世界很大，单节很虚。

### 错误 5：只做内容，不沉淀模板

结果是团队一直在消耗创意，而不是积累资产。

## 12. 我会先落地的组织动作

如果现在由我直接开干，我第一周只做这些事：

1. 定一页北极星
2. 定“一节”的标准定义
3. 定八个功能位
4. 定八张卡模板
5. 定技术集成 Owner 的闸门职责
6. 定 Codex 的最小输入格式
7. 做出一节灰盒样板
8. 用这节样板完成第一次模板沉淀

我不会第一周就去扩章节、扩世界、扩角色库。

## 13. 一张最简流程图

公司级最简流程应当是：

战略定义 -> 单节立项 -> 八角色出规格 -> 技术集成审查 -> Codex / 开发落地 -> 灰盒试玩 -> CEO / Owner 验收 -> 模板沉淀 -> 下一节复用

这条链一旦稳定，后面再叠：

- 世界观
- 季度规划
- 增长体系
- 商业模式
- AI 个性化

才是往上长，而不是往旁边散。

## 14. 最终结论

如果我是 CEO，我不会从“该设哪些部门”开始。

我会从“这家公司最小的生产闭环是什么，怎样让它每周都稳定发生，而且下一次比上一次更便宜、更快、更稳”开始。

一句话总结：

先把单节生产线做硬，再谈公司规模；先把流程主链做出来，再谈组织扩张；先把拍板权和产物接口切清，再让人进入系统。
