# 周运营 SOP

版本：v0.1  
定位：把“任务冒险引擎”的单节工业化生产，变成团队每周可重复执行的固定节奏

## 1. 适用范围

本 SOP 适用于当前阶段的工作对象：

- 单节灰盒原型
- 单节正式原型
- 单节改版
- 错误系统补完
- 单节交互重构

不适用于：

- 纯世界观脑暴
- 没有系统对象的概念讨论
- 没有成功标准的开放探索

## 2. 本周运营目标

每周运营系统只追求四件事：

1. 有且只有一到两个单节在主生产线上推进
2. 每个进入生产线的单节都有完整规格
3. 每周至少完成一次“灰盒落地 -> 试玩 -> 复盘”
4. 每周模板库都有新增资产，不允许只消耗创意

## 3. 周节奏总览

| 日期 | 会议 / 动作 | 目标 | 主责 |
|---|---|---|---|
| 周一上午 | 立项与规格审查会 | 决定本周做哪一节，规格是否成立 | CEO / 产品总负责人 |
| 周二上午 | 实现拆解会 | 切任务、定复用、定技术红线、生成 Codex 输入包 | 技术集成 Owner |
| 周三至周四 | 灰盒实现 | 跑通状态、规则、报错、重跑闭环 | 技术集成 Owner + Codex / 工程 |
| 周五上午 | 试玩验证会 | 看孩子行为、看家长证据、看关键问题 | 模板沉淀官 |
| 周五下午 | 复盘沉淀会 | 沉淀模板、记录技术债、决定下周优化 | 模板沉淀官 + 技术集成 Owner |

## 4. 周一：立项与规格审查会

### 4.1 会议目标

只回答一件事：

这节是否值得进入本周生产线。

### 4.2 必备输入

- 节目标卡
- 系统结构卡草稿
- 成功标准草稿
- 不做项清单

如果这些都没有，不开会，直接打回补文档。

### 4.3 必问问题

1. 这节主打哪 1-2 个主轴
2. 这节的系统对象是什么
3. 孩子离开时应带走什么方法
4. 家长端最终能看到什么证据
5. 本节明确不做什么
6. 成功标准是否能被验收

### 4.4 决策结果

只能输出三种结果：

- 通过
- 缩减后通过
- 打回重写

### 4.5 会后产物

- 最终版节目标卡
- 立项结论
- 本周版本边界

## 5. 周二：实现拆解会

### 5.1 会议目标

把“可做的一节”拆成“可执行的一节”。

### 5.2 必备输入

- 节目标卡
- 进入钩子卡
- 系统结构卡
- 错误设计卡
- 交互流程卡
- 主轴校验卡

### 5.3 必问问题

1. 哪些结构复用旧骨架
2. 哪些模块必须新建
3. 哪些地方允许临时 hardcode
4. 哪些需求必须砍掉，否则会破坏本周节奏
5. Codex 的最小输入包是否齐了

### 5.4 决策结果

- 任务拆解清单
- 技术红线
- Codex 输入包
- 负责人分配

### 5.5 不允许发生的事

- 会后还不知道谁是第一责任人
- 会后还不知道哪些内容本轮不做
- 会后还要靠口头给 Codex 补规格

## 6. 周三到周四：灰盒实现

### 6.1 阶段目标

先把核心闭环跑通，不追求包装完整。

### 6.2 必达结果

1. 状态能跑
2. 规则能改
3. 至少两类错误能暴露
4. 至少一次重跑成立
5. 家长证据结构能生成

### 6.3 技术集成 Owner 每天检查四件事

1. 状态命名是否统一
2. 错误类型是否统一
3. 法则台是否复用统一逻辑
4. 是否出现了会破坏长期骨架的临时实现

### 6.4 Codex / 工程每日回传格式

每天回传只讲五项：

1. 已完成什么
2. 哪条状态流已跑通
3. 哪条错误流已跑通
4. 还卡在哪个分歧点
5. 是否有需要拍板的问题

## 7. 周五上午：试玩验证会

### 7.1 会议目标

验证这节是否真的在训练目标能力，而不是只“看起来成立”。

### 7.2 必备输入

- 当前灰盒版本
- 试玩记录
- 观察笔记
- 家长证据草稿

### 7.3 必问问题

1. 孩子有没有真实发生判断
2. 孩子是不是只在试错，不在缩小范围
3. 报错有没有帮助孩子更接近问题
4. 家长能不能一句话说出这节练了什么
5. 这节有没有值得复用的结构

### 7.4 通过标准

至少满足以下四条：

1. 孩子完成至少一次真实问题归类判断
2. 孩子完成至少一次“改 -> 跑 -> 看报错 -> 再改”
3. 至少两类错误能被观察到
4. 家长证据可以被清楚复述

## 8. 周五下午：复盘沉淀会

### 8.1 会议目标

把本周经验变成资产，而不是停留在口头感受。

### 8.2 必答问题

1. 这节属于哪种节类型
2. 哪些模块可复用
3. 哪类错误模板最好用
4. 哪一步最卡
5. 哪些坑以后必须避免
6. 下周如果做相似内容，哪些东西不用重做

### 8.3 会后产物

- 模板归档卡
- 模板库更新项
- 技术债清单
- 下周建议

## 9. 每周必须留下的文档

每周结束后，至少应新增或更新以下文档：

1. 本周通过的节目标卡
2. 本周 Codex 输入包
3. 本周试玩问题单
4. 本周模板归档卡
5. 本周技术债清单

如果这些东西没有留下，说明本周只有开发动作，没有组织沉淀。

## 10. 当前阶段不该开的会

为了防止团队节奏被打散，当前阶段应减少以下会议：

- 无明确输出的脑暴会
- 不带文档输入的讨论会
- 把所有人都拉上的大同步会
- 纯汇报式站会

会议不是为了“大家都知道了”，而是为了让关键事项被拍板、被交接、被推进。

## 11. 红线

出现以下任一情况，应立刻停下来纠偏：

1. 没有成功标准就开始实现
2. 没有错误系统就开始做大包装
3. 每一节都重新定义状态和错误命名
4. 试玩结束后没有进入模板沉淀
5. CEO 持续陷入交互细节拍板，说明中层功能位失效

## 12. 当前阶段 CEO 每周只盯什么

CEO 每周只需要强盯四个指标：

1. 本周有没有单节闭环真正跑通
2. 第二节是否比第一节更容易做
3. 报错系统有没有更统一
4. 模板库有没有新增资产

如果这四个指标持续改善，公司就在形成真正的生产能力。

## 13. 最终原则

周运营 SOP 的价值，不是把团队管理得更重，而是让每周都稳定发生同一种高质量推进。

一句话总结：

周一定方向，周二切执行，周三周四跑灰盒，周五做验证和沉淀；每周都要有产出、有拍板、有复盘、有资产。
