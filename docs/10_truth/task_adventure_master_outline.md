# 任务冒险引擎总纲

版本：v0.1 整理版  
整理日期：2026-03-25

## 1. 一句话定义

任务冒险引擎不是一门更花哨的儿童编程课，也不是披着学习外衣的小游戏，而是一种把任务冒险、角色身份、系统反馈、思维训练和家长可见成长证据合成在一起的儿童互动成长系统。

在当前阶段，这一系统进一步升级为一种以“对话驱动”为主要交互介质的儿童编程思维体验。

它的核心目标是：

- 让孩子在愿意持续进入的世界里，自然调用五种核心思维能力
- 让家长看到的不是课时和正确率，而是可解释、可追踪、可累积的成长证据
- 让团队生产的不是一次性 demo，而是可复用、可扩展、可工业化的单节系统骨架

## 2. 它是什么，不是什么

### 它是什么

- 一种以任务局面驱动参与的成长媒介
- 一种以世界感和身份感承载编程思维训练的产品形态
- 一种把孩子端体验、系统端逻辑、家长端证据放在同一条任务链上的业务系统
- 一种让孩子通过与世界、角色、系统持续对话来完成建模、调试和修正的对话驱动式编程体验

### 它不是什么

- 不是代码语法启蒙课
- 不是老师讲、孩子听、做题、交作业的线上课变体
- 不是以剧情、动画、奖励为主驱动的轻娱乐产品
- 不是低幼化哄学产品

## 3. 项目真正训练的能力

项目表面上讲“编程思维”，底层训练的是面对复杂问题时的系统化行动能力。

五大主轴如下：

1. 定义问题：看清目标、条件、约束、缺失信息和真正卡点
2. 拆解任务：把复杂局面拆成模块、顺序、依赖和关键路径
3. 建立规则：把主观想法转成系统可执行的条件、流程和例外处理
4. 调试纠错：在结果与预期不一致时，通过观察、猜测、验证和修改推进理解
5. 抽象迁移：从一次任务提炼出一类问题的共性结构和方法

## 4. 目标用户与双边价值

### 面向孩子

目标年龄为 7-13 岁。产品要给孩子的不是“学了什么知识点”，而是以下体验：

- 我被当成一个有能力的人
- 这里有任务需要我介入
- 我真的把复杂事情看懂了
- 我设计的规则让世界发生了变化
- 出错后我不是被否定，而是可以继续修
- 我开始发现不同任务背后的共同规律
- 我和以前不一样了，我在变强

### 面向家长

家长端要看到的不是传统学习指标，而是成长证据：

- 孩子如何定义问题
- 孩子如何拆解结构
- 孩子如何表达规则
- 孩子如何面对错误并继续调试
- 孩子是否开始出现方法迁移

项目的续费逻辑不是“孩子开心”本身，而是“孩子愿意回来 + 家长能看见真实成长”同时成立。

## 5. 业务本体：最小生产单元是“一节”

项目当前最重要的单位不是章、季或世界观，而是“一节”。

一节的定义：

- 时长约 20-30 分钟
- 是一个孩子可以完成的最小认知闭环
- 不是剧情片段、知识点或练习题

一节必须包含的最小循环：

1. 进入异常
2. 判断真正问题
3. 看清系统结构
4. 修改法则或规则
5. 运行系统
6. 观察可观测错误
7. 局部修复并重跑
8. 带着一条可迁移的方法离开

如果缺少重跑和报错，这更像剧情解谜。  
如果缺少提炼，这更像一次性任务。  
如果缺少判断问题，这更像教程操作。

## 6. 产品结构模型

项目不是一堆页面，而是五层嵌套系统：

1. 世界层：给任务提供长期意义、连续性和归属感
2. 身份层：让孩子以参与者而不是学生的姿态进入
3. 任务层：以局面触发真实介入，而不是以题目触发点击
4. 思维层：五大主轴在行动中被调用并升级
5. 证据层：把过程痕迹沉淀成孩子、家长和系统都可读取的成长记录

三端本质上是同一件事的不同投影：

- 孩子端：世界、任务、规则、反馈、成长感
- 家长端：过程痕迹、能力变化、下一步方向
- 系统端：状态、错误、支架介入、成长画像、推荐逻辑

## 7. 孩子端的核心体验循环

第一版必须跑通的不是大世界，而是以下主循环：

看异常 -> 判问题 -> 拆模块 -> 改法则 -> 跑系统 -> 看报错 -> 再修一次 -> 得到模式句

在 `Vibe Coding for Kids` 已升级为一级战略之后，这条主循环的主要交互介质不再默认是题目、点选或表单，而应逐步升级为：

对话表达 -> 系统理解 -> 规则修改 -> 运行反馈 -> 对话修正

对应的七段体验节奏：

1. 进入异常：先让孩子想管这件事
2. 判断真问题：区分表象和真正问题
3. 拆解系统：把局面变成模块和路径
4. 修改法则：真正改条件、顺序、权限或路径
5. 运行系统：看到自己的设计影响世界
6. 读取报错并重跑：形成真实调试体验
7. 母纹提炼：带走可迁移的方法，而不是只停在“通关”

## 8. 三个核心界面

第一版至少要成立三个孩子端核心界面：

1. 异常现场界面
   - 负责世界感、进入感、情绪钩子
   - 呈现场景异常、角色反应、世界回响和初步线索
2. 系统视图界面
   - 把局面变成可分析对象
   - 呈现模块、状态、连接、依赖和异常位置
3. 法则台界面
   - 让孩子修改规则、运行系统、查看报错、进入二次修复
   - 呈现可改规则、运行按钮、反馈、回放和重跑入口

## 9. 错误系统是产品灵魂

这个项目能否成立为编程思维产品，关键不在剧情，而在错误系统。

错误系统的核心原则：

- 任何失败都必须可观测
- 错误必须有类型，不能只是统一失败
- 错误必须局部可见，不能整节黑箱
- 系统必须允许“假设 -> 修改 -> 重跑”
- 剧情负责让我想修，诊断系统负责让我会修

第一版建议统一四类错误：

- 条件错误
- 顺序错误
- 权限错误
- 状态 / 边界错误

错误语言必须双层存在：

- 底层：真正可诊断的逻辑错误模型
- 表层：孩子可感知的世界语言

示例：

- `SequenceMismatch` -> 纹路闭合顺序错误
- `ConditionMismatch` -> 火脉未满足开启条件

## 10. 家长端真正要交付的东西

家长端不能只显示：

- 完成了哪一节
- 用了多久
- 是否通关

第一版至少要沉淀三类证据：

1. 问题定义证据：孩子是否区分了表象与真因
2. 结构拆解证据：孩子是否拆出了模块、顺序和依赖
3. 调试证据：孩子是否在错误后继续定位、修改和重跑

这些证据比正确率更接近项目真正的教育价值。

## 11. AI 的正确位置

AI 不是主角，也不能代做。

AI 适合出现在以下思维尚未稳定的节点：

- 理解节点
- 拆解节点
- 调试节点
- 迁移节点

AI 的职责是：

- 提问
- 镜像
- 提示
- 陪同

AI 不应该：

- 直接给出完整答案
- 跳过孩子的核心判断
- 替孩子完成关键规则设计
- 把调试过程变成自动修复

在 `Vibe Coding for Kids` 战略下，AI / NPC 的正确位置进一步明确为：

- 作为对话驱动体验中的提问者、镜像者、缩小范围者和结构翻译者
- 负责帮助孩子把自然语言表达转成更清楚的系统判断和修改动作
- 不能把对话做成答案发放器

## 12. 工业化生产方法

项目要成立，不能只靠灵感做内容，必须先证明“单节可以稳定生产”。

### 12.1 八个固定工位

任何一节都应经过以下工位：

1. 节目标定义
2. 戏剧钩子设计
3. 系统建模
4. 错误反馈设计
5. 孩子交互路径设计
6. 认知校验
7. 家长证据提炼
8. 复盘与模板沉淀

这 8 个工位本质上是 8 个功能位，不是必须对应 8 个自然人。

- 一个人可以兼任多个工位
- 但这 8 种功能不能消失
- 任何单节立项、开发、试玩、复盘，都应经过这 8 个功能位的检查

### 12.2 八角色功能位说明

八个工位对应的是单节工业化开发中的八种核心职责：

1. 节目标官：定义这节为什么存在，主打哪 1-2 个主轴，孩子离开时带走什么方法，家长看见什么证据
2. 戏剧钩子官：定义孩子为什么一进来就想管这件事，负责开场异常、第一印象和情绪色调
3. 系统建模官：把故事压成系统，明确模块、输入输出、条件、状态、依赖和搅局点
4. 错误反馈官：把“系统出错”设计成可观测、可分类、可定位、可调试的反馈
5. 交互路径官：定义孩子先看什么、先判断什么、能改什么、何时运行、何时重跑
6. 认知校验官：防止整节变成伪编程思维，检查主轴、判断、调试和认知抬升是否真实发生
7. 家长证据官：把孩子真实发生的行为翻译成家长能信、能懂、能复述的成长证据
8. 模板沉淀官：把一节做完后的可复用结构、错误模板、踩坑清单和优化建议沉淀进模板库

这 8 个角色的核心意义，不是分工更细，而是防止以下问题反复出现：

- 只有剧情，没有系统
- 只有异常，没有错误模型
- 只有操作流程，没有真实判断
- 只有孩子端体验，没有家长端证据
- 每做完一节，下一节仍然从零开始

### 12.3 八角色对应的八张关键卡

为了让功能位真正落地，每个工位都应至少产出一张关键卡：

1. 节目标官 -> 节目标卡
2. 戏剧钩子官 -> 进入钩子卡
3. 系统建模官 -> 系统结构卡
4. 错误反馈官 -> 错误设计卡
5. 交互路径官 -> 交互流程卡
6. 认知校验官 -> 主轴校验卡
7. 家长证据官 -> 家长证据卡
8. 模板沉淀官 -> 模板归档卡

从产品方法论上看，这 8 张卡不是文档负担，而是把“模糊创意”压缩成“可协作、可复盘、可执行对象”的中间层。

### 12.4 规格包驱动执行

Codex 不应直接接收愿景式需求，而应只接收结构化规格包。

规格包最少包含四张核心卡：

1. 系统结构卡
2. 错误设计卡
3. 交互流程卡
4. 成功标准卡

建议补充三张辅助卡：

5. 节目标卡
6. 世界语言与文案约束卡
7. 明确不做项清单

### 12.5 技术集成 Owner

这个角色不是最会写代码的人，而是把每一节从一次性 demo 变成同一种系统的人。

第一版要盯住五个统一层：

- 状态系统统一
- 错误类型统一
- 法则台组件统一
- 运行与重跑机制统一
- 配置格式统一

这个角色的本质工作不是把当前一节做完，而是让下一节不必重新发明。

## 13. 第一版优先级

第一版的优先顺序应当是：

1. 单节循环跑通
2. 错误系统统一
3. 法则台统一
4. 配置格式统一
5. 技术债可见

先做灰盒，后做包装。

第一轮只看：

- 状态是否能跑
- 规则是否能改
- 错误是否能出
- 重跑是否成立

第二轮再补：

- 世界文案
- 视觉包装
- 动效表现
- 更完整的神话演出

## 14. 设计气质要求

视觉和交互上，这个项目希望呈现的不是标准 SaaS 后台，也不是廉价卡通，而是一种克制、编辑式、有世界感的高质量体验。

设计上应坚持：

- 不低幼
- 不后台化
- 不密集堆砌
- 不依赖硬边框和常规卡片模板
- 强调留白、层次、节奏和身份感

但这类设计意图必须继续被翻译成硬规则、组件合同、反例和验收标准，才能被 agent 稳定执行。

## 15. 第一版成功标准

第一版是否成立，先回答四个问题：

1. 孩子会不会愿意把一节玩完
2. 孩子会不会在没有直接答案的情况下，真实做出一次“判断 -> 修改 -> 重跑”
3. 报错反馈是不是让孩子更接近问题，而不是更困惑
4. 家长看完后能不能一句话说出“这节练的是哪种思维能力”

只要这四条成立，第一版就是成功的。

## 16. 当前项目的核心结论

这个项目当前最重要的工作，不是继续扩世界观，也不是堆更多课程内容，而是把“一节”做硬、做清楚、做成可复制的方法。

一句话总结：

先定义业务本体，再生长产品形态；先跑通单节闭环，再扩章节宇宙；先把模糊愿景压成可执行规格，再交给 Codex 实现。
