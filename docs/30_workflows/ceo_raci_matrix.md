# CEO RACI 责任矩阵

版本：v0.1  
定位：明确单节工业化生产中，谁负责、谁拍板、谁参与、谁被同步

## 1. 说明

RACI 含义如下：

- `R` = Responsible，直接负责产出
- `A` = Accountable，最终拍板负责
- `C` = Consulted，必须参与讨论
- `I` = Informed，需要被同步

原则：

- 每个关键事项只能有一个 `A`
- 可以有多个 `R`
- 如果没有 `A`，事情一定会悬空
- 如果所有人都是 `C`，组织就会变成讨论会

## 2. 角色定义

本矩阵使用以下角色：

1. CEO / 产品总负责人
2. 世界观 / 内容总负责人
3. 节目标官
4. 戏剧钩子官
5. 系统建模官
6. 错误反馈官
7. 交互路径官
8. 认知校验官
9. 家长证据官
10. 技术集成 Owner

## 4. 几个最关键的 A 必须钉死

## 5. 几个典型冲突，谁说了算

### 冲突 1：故事很美，但系统建不起来

谁说了算：

- 系统能不能运行，由系统建模官和技术集成 Owner 说了算
- 是否继续做这节，由 CEO / 产品总负责人拍板

### 冲突 2：体验很顺，但认知动作很假

谁说了算：

- 认知校验官有一票否决权

### 冲突 3：Codex 很快能做，但会破坏长期统一性

谁说了算：

- 技术集成 Owner 说了算

### 冲突 4：孩子端成立，但家长端说不清价值

谁说了算：

- 家长证据官负责重写表达
- 是否因此延期上线，由 CEO / 产品总负责人拍板

### 冲突 5：多个产品路径都讲得通

谁说了算：

- 先由技术集成 Owner 和节目标官整理方案差异
- 最终由 CEO / 产品总负责人拍板默认路径

## 7. 当前阶段最适合的小团队兼任方案

如果团队还小，可以这样兼任，但逻辑角色仍然保留：

| 人员类型 | 建议兼任功能位 |
|---|---|
| CEO / 产品负责人 | CEO / 产品总负责人 + 节目标官 |
| 内容负责人 | 世界观 / 内容总负责人 + 戏剧钩子官 + 家长证据官 |
| 系统策划 | 系统建模官 + 错误反馈官 + 认知校验官 |
| 交互负责人 | 交互路径官 |
| 技术负责人 | 技术集成 Owner + 模板沉淀官 |
| Codex / 工程 | Codex / 工程实现 |

这只是兼任方式，不改变拍板边界。

## 8. RACI 怎么真正用起来

这张表不是贴在墙上好看，而是每次开会前都要做两件事：

1. 先确认本次议题的 `A` 是谁
2. 再确认会后要交付的产物是什么

只要这两件事没有先写清，会议大概率会变成意见交换，而不是决策动作。

## 9. 最终原则

RACI 的价值，不是把团队管得更重，而是让每一个关键结果都有人真正兜底。

一句话总结：

公司不是靠“大家都很努力”跑起来的，而是靠“关键事项始终有人负责、有人拍板、有人交付”跑起来的。
